# 数据方案 — 3 选 1

> **核心问题**: 招牌 → 店名识别是真,但"4 平台同套餐价格 + 销量 + 标注"怎么来?
> 这是端侧 AI 创新挑战赛,**重点是端侧 VLM 识别招牌**,团购数据怎么落地决定 demo 真实感 + 合规风险。

---

## 方案 A · 纯 Mock(当前)— 0 改动,2 分钟可用

**做法**: `MockDeals.kt` 写死 7 店 × 5 平台(抖音/美团/美团外卖/淘宝闪购/京东秒送)。

**优点**:
- ✅ **零依赖、零爬取、零法律风险** — 完全合法
- ✅ 离线 100% 可用,演示最稳(无网也跑)
- ✅ 价格可控可调,方便 demo 突出"省 X 元"对比

**缺点**:
- ❌ 评委看破:数据写死,不真实
- ❌ 街拍真实店(比如路边"兰州拉面")会查不到
- ❌ 创意分打折(没真实数据接入)

**评委视角**: "MVP 演示 OK,但生产化不够"

---

## 方案 B · 半真实(招牌+定位真,套餐从地图 API)— 半天工

**做法**: 招牌识别真 + GPS 真,**套餐数据走开放 API**:
- **高德 / 百度地图 POI API** — 拿店名/地址/评分/经纬度(免费层 5000 次/天)
- **美团/大众点评开放平台** — 不开放,但有 **Web 搜索页** 可爬(robots.txt 限制)
- **淘宝/京东无开放 API** — 走 H5 搜索页 URL 跳转,价格显示为"实时获取"

**代码改动**:
```
新增 ApiDeals.kt (用 OkHttp + 高德 POI Key)
MockDeals.findInfo() 改为 ApiDeals.findInfo()
  - 命中本地 cache: 直接返回 MockDeals
  - miss: 网络请求高德 → 缓存 → 返回
```

**优点**:
- ✅ 评分/地址/距离真实(高德)
- ✅ 走 App 跳转获取价格(合法:deeplink 打开美团/抖音/淘宝官方 App 搜索结果)
- ✅ 离线降级到 MockDeals cache

**缺点**:
- ⚠️ 需要高德 API Key(注册 5 分钟,免费够用)
- ⚠️ "销量/标注" 字段仍得编(API 不会返回)
- ⚠️ 网络延迟 200-500ms,UI 要 skeleton

**评委视角**: "有真实数据接入,知道工程化边界" ⭐ 推荐

---

## 方案 C · 真实爬取 — 1-2 天工,**法律风险高**

**做法**: 写爬虫爬 4 平台公开搜索页(美团/抖音/淘宝/京东)。

**优点**:
- ✅ 完全真实价格 + 销量 + 标注
- ✅ 演示最有冲击力

**缺点**:
- ❌ **违法**: 违反《反不正当竞争法》 + 平台 ToS,2024 美团/抖音都告过爬虫公司
- ❌ 反爬: 验证码 / IP 风控 / WebView 加密 / JS 动态渲染
- ❌ 4 平台 × 7 店 × 5 套餐 = 140 个 SKU,每次搜索返 1000+ 条,得写分布式爬虫 + 去重
- ❌ 比赛演示有 IP ban 风险

**评委视角**: "技术牛,但商业落地合规有问题" — 风险大

---

## 我的推荐

**方案 B**(半真实)— 高性价比,符合赛事"端侧 + 云端协同"主题:
1. **招牌识别 100% 端侧**(MNN + Qwen3-VL-2B)— 核心创新点
2. **GPS + 高德 POI 真实**(API Key 注册 5 分钟,免费层够用)
3. **4 平台价格走 deeplink 跳官方 App**(合法,且"实时"语义真实)
4. **销量/标注字段编合理值**(写明"预估销量")

代码改动量预估:
- 新增 `ApiDeals.kt` (150 行, OkHttp + 缓存)
- 新增 `AmapProvider.kt` (80 行, 高德 POI 接口)
- `MockDeals.kt` 改成 cache + 兜底
- 视频脚本改:"招牌本地识别 → 网络拉真实店信息 → 跳 4 App 比价"

**耗时**: 4-6 小时(含注册 API Key + 写代码 + 调试)
