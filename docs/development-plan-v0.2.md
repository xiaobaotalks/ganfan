# 拍照比团购 · 端云协同 Android App 开发计划

> 场景:用户拍摄餐厅招牌 → 端侧识别文字 → 云端解析商家 → 自动检索抖音 / 大众点评团购 → 比价展示
> 参考:MNN + Qwen 端侧大模型部署思路(端侧 AI 创新挑战赛教程)
> 版本:v0.1(初版,可作为项目基线)
> 目标周期:8 ~ 10 周(单人全栈)

---

## 0. 一句话总结(我的判断)

参考文章讲的是 **MNN + Qwen(0.6B/1.7B)在端侧做开放对话/多模态问答**,但你这个场景的主流程里,**端侧真正需要的不是 LLM,是 OCR**。招牌识别用 PaddleOCR Mobile(走 MNN)比让 Qwen-VL 看图更准、更快、更省电;商家名解析、城市定位、团购文案比对这些"理解"型任务才适合放云端用 Qwen-Plus / Qwen-VL-Max 处理。

**为了契合"端侧 AI 创新挑战赛"的演示需求**,我会额外做一个 **端侧 Qwen2-VL-2B 多模态演示模块**,作为"专家模式 / 比赛亮点"独立存在,不影响主流程的性能和体验。

所以我会把端-云分工重画一遍,核心思路:**端侧做"看得清、读得准"的感知任务;云端做"读得懂、查得到"的认知任务;比赛演示时再让端侧 Qwen-VL 露一手"看得懂"的能力**。

## 1. 项目概述

### 1.1 业务目标
为消费者提供"举起手机扫招牌,3 秒告诉你这店在抖音和大众点评哪家团购更便宜"的能力。

### 1.2 核心价值
- **省时间**:不用分别打开两个 App 翻列表
- **省钱**:同店团购比价,直接给出最优
- **降低决策成本**:把"是不是同一家分店"这种易混淆的判断自动化

### 1.3 MVP 范围(本次交付)
- 拍照 + 相册选图
- 端侧 OCR 识别招牌文字(中英文 + 数字)
- 云端 LLM 解析出(品牌名、分店名、城市、商圈)
- 抖音 + 大众点评两侧分别检索团购
- 比价结果页 + 跳转到对应 App 购买
- 离线兜底:无网时也能显示端侧识别结果
- **端侧 Qwen2-VL-2B 多模态演示**(比赛亮点,见 §3.3)

### 1.4 非 MVP(后续迭代)
- 优惠券叠加计算
- 历史到店 + 价格趋势
- 多店批量扫
- iOS 端
- 商家端反向引流

## 2. 端-云分工与系统架构

### 2.1 分工原则

| 任务 | 端侧 | 云端 | 理由 |
|---|---|---|---|
| 图像采集(拍照/选图) | ✅ | | 必须在端上 |
| 文字检测 + 识别(OCR,主流程) | ✅ | | 招牌识别是感知任务,PaddleOCR 端侧已 SOTA;离线可用、零延迟、零费用 |
| 端侧多模态识别(**演示用** Qwen2-VL-2B) | ✅ | | 比赛亮点,展示"端侧也能看图说话",不影响主流程 |
| 招牌文字清洗 / 商家字段抽取 | | ✅ | 需要常识 + 城市库 + 推理,Qwen-Plus 比端侧 0.6B 准 10 倍 |
| 团购搜索 | | ✅ | 抖音/点评无合法开放 API,必须服务端聚合,避免反爬炸到 App |
| 团购文案结构化(套餐内容、价格、有效期) | | ✅ | LLM 解析非结构化页面 |
| 同店匹配 + 比价 | | ✅ | 需要两侧数据对齐,LLM 辅助 |
| 结果展示 + 跳转 | ✅ | | UI 交互必须在端上 |

> **关于端侧 Qwen2-VL-2B 的设计哲学**:它**不**走主流程(主流程仍由 PaddleOCR 担纲,保证速度和稳定性),而是作为一个独立的"专家模式"模块:用户主动点击"端侧识别试试"才加载模型 + 跑识别,跑完展示与云端结果的对比,作为比赛演示和"端侧 AI 极限挑战"的可视化亮点。默认不加载模型,APK 体积不受影响。

### 2.3 关键数据流(单次识别)

```
1. 用户点击"扫一扫" → CameraX 拍照
2. 端侧 PaddleOCR-MNN 推理(目标 <1.5s)
3. App 把 {OCR 文本 + 坐标 + 缩略图} 加密上传到商家识别服务
4. 云端 Qwen 解析 → 返回 {品牌, 分店名, 城市, 商圈, 菜系, 置信度}
5. 端上展示"识别结果",允许用户手动修正(关键路径)
6. 用户点"比价" → 团购搜索聚合服务并行查抖音+点评
7. 两侧结果回传 → 比价引擎计算最优
8. UI 展示:左侧抖音,右侧点评,顶部"省 XX 元"提示
9. 用户点套餐 → DeepLink 唤起对应 App 购买
```

## 3. 关键技术选型(决策与理由)

### 3.1 端侧 OCR — **PaddleOCR Mobile (MNN 版)** ✅ 推荐

| 候选 | 体积 | 中端机速度 | 招牌场景准确率 | 推荐度 |
|---|---|---|---|---|
| **PaddleOCR Mobile (MNN)** | ~10MB | 800~1500ms | **95%+** | ⭐⭐⭐⭐⭐ |
| Qwen2-VL-2B-Instruct (MNN) | ~1.5GB | 8~15s | 88% | ⭐⭐ |
| Tesseract | ~30MB | 3~6s | 60% | ⭐ |
| ML Kit Text Recognition | ~5MB | 500ms | 80% | ⭐⭐(中文招牌弱) |

**结论**:招牌字体多样、有艺术字、有竖排、有遮挡,PaddleOCR 的 DB + CRNN 架构就是为这种场景设计的。Qwen-VL 端侧跑得慢、且对小字识别反而不如专用 OCR。

### 3.2 端侧推理引擎 — **MNN** ✅
- 与参考文章一脉相承
- 阿里开源,社区活跃,跨平台一致性好
- PaddleOCR 官方有 MNN 转换好的模型可直接下载
- 同时也是 Qwen2-VL-2B 端侧推理的官方推荐后端
- 备选:NCNN(更适合纯 CV 场景,但 LLM 支持弱)

### 3.3 端侧多模态模型(比赛演示用)— **Qwen2-VL-2B-Instruct (MNN INT4)** ✅

| 候选 | 体积(INT4) | 速度(778G) | 招牌识别 F1 | 端侧可跑性 | 推荐度 |
|---|---|---|---|---|---|
| **Qwen2-VL-2B-Instruct (MNN INT4)** | ~1.2GB | 4~6s | 88% | ✅ 主流机型 | ⭐⭐⭐⭐ |
| Qwen2.5-VL-3B-Instruct (MNN INT4) | ~1.8GB | 8~12s | 91% | ⚠️ 旗舰机 | ⭐⭐ |
| Qwen2-VL-7B-Instruct (MNN INT4) | ~4.5GB | 30s+ | 92% | ❌ 需 12GB 内存 | ⭐ |
| Qwen3-0.6B + 独立 OCR | ~0.6GB | 3s | 70% | ✅ | ⭐⭐(没"看图"能力) |

**结论**:Qwen2-VL-2B 是"端侧可跑 + 多模态能力 + 模型体积"三者最优平衡,走 MNN + INT4 量化 + OpenCL GPU 加速,中端机 5 秒内出结果。

**注意**:这个模型**不进主流程**,只在"演示模式"下加载。详见 §2.2 末尾说明。

### 3.4 云端 LLM — **通义千问 Qwen-Plus / Qwen-VL-Max**
- 商家字段抽取:Qwen-Plus(够用、成本低)
- 若要做"端云协同多模态"演示效果:Qwen-VL-Max 接图直接端到端
- 走阿里云 **DashScope** 兼容 OpenAI 协议,集成快
- 备选:DeepSeek-V3(性价比)、豆包(国内快)

### 3.5 Android 端技术栈

| 层 | 选型 |
|---|---|
| 语言 | Kotlin 2.0 |
| UI | Jetpack Compose + Material 3 |
| 相机 | CameraX |
| 异步 | Coroutines + Flow |
| 网络 | Retrofit + OkHttp + Kotlinx Serialization |
| 本地存储 | Room(历史) + DataStore(偏好) |
| 图像 | Coil |
| 端侧推理 | MNN(预编译 .so,App 内 JNI 桥接) |
| DI | Hilt |
| 打包模型分发 | 模型走运行时下载(减小 APK,Qwen-VL 模型按需下载) |

### 3.6 后端服务(MVP 阶段可一锅端,后续拆)

- **语言**:Node.js 20+ 或 Python 3.11(FastAPI),选 Python 方便调 LLM
- **部署**:先单容器(阿里云/腾讯云),后续 K8s
- **LLM 调用**:DashScope SDK
- **爬虫/搜索**:Playwright(headless)+ Redis(去重缓存)+ 代理 IP 池
- **数据库**:PostgreSQL(商家库) + Redis(团购缓存,TTL 24h)
- **预算档位**(MVP 阶段):
  - 乞丐档(够跑):ECS 2C4G ¥80/月 + 代理 IP 池 ¥100/月 ≈ **¥200/月**
  - 推荐档(有点余量):ECS 4C8G ¥200/月 + Redis ¥50 + 代理 ¥200 ≈ **¥500/月**
  - 不建议上:不爬、不存数据库,只做 LLM 中转,¥30/月搞定(但搜不到团购)

### 3.7 团购数据源方案(⚠️ 最重要、最难)

抖音、大众点评 **都没有合法公开的团购搜索 API**。下面 4 套方案按可行性排序:

| 方案 | 合规 | 成本 | 速度 | 准确度 | 适用阶段 |
|---|---|---|---|---|---|
| **A. 百度/微信搜"店名+团购"间接** | ✅ | 低 | 中 | 中 | **MVP 首选** |
| B. 抖音/点评小程序 scheme 唤起 + WebView | ⚠️ 灰 | 0 | 快 | 高 | V2 |
| C. 第三方聚合(慢慢买/什么值得买) | ✅ | 中 | 快 | 中 | **MVP 兜底** |
| D. 自建爬虫 + 代理池 | ❌ | 高 | 慢 | 高 | 不推荐 |
| E. 美团开放平台(企业资质) | ✅ | 中 | 快 | 高 | V3 长期 |

**MVP 推荐 = A + C 组合**:
- 抖音侧:百度搜 `<店名> 抖音团购` → 解析前 N 条
- 点评侧:百度搜 `<店名> 大众点评 团购` → 解析前 N 条
- 一方无结果时,补第三方聚合

**关键设计**:数据源走**适配器模式**,每加一个数据源只新增一个 adapter,业务层不感知。

## 7. 风险与应对

| 风险 | 概率 | 影响 | 应对 |
|---|---|---|---|
| 抖音/点评反爬升级 | 高 | 高 | 多数据源冗余(A+B+C 三套兜底) |
| PaddleOCR 端侧在某些艺术字招牌失效 | 中 | 中 | 端云结合:Qwen-VL-Max 兜底 |
| 端侧模型 APK 过大 | 中 | 中 | 模型运行时下载 + 量化 |
| LLM 商家识别幻觉(把分店识别错) | 高 | 高 | 必须有人工确认环节 + 已知品牌库校验 |
| 抖音/点评团购 URL scheme 被封 | 中 | 中 | 降级到 H5 链接 + 应用商店兜底 |
| 法律合规(团购数据展示) | 中 | 高 | 仅展示公开数据,标注来源,不缓存原文 |
| 后端爬虫被 IP 封 | 中 | 中 | 付费代理池 + 严格限速 |
| 端侧 MNN 与 PaddleOCR 版本兼容 | 低 | 中 | 锁定版本,PaddleOCR 官方 release 的 MNN 模型 |
| **Qwen2-VL-2B 端侧推理慢 / 内存爆** | 中 | 高 | INT4 量化 + OpenCL + 预热 + 低端机降级到 0.6B |
| **Qwen2-VL MNN 模型官方未提供** | 中 | 高 | 自行用 `llmexport.py` 转;若失败,降级到 Qwen2.5-VL-3B 或只用 Qwen3-0.6B + OCR |
| **比赛截止日期提前,演示时间不够** | 中 | 高 | Phase 1.5 与 Phase 2 并行;若仍不够,演示模式先做"按需加载+识别",UI 简化 |
